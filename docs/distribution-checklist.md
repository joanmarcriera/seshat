# Distavo — Distribution & Notarization Checklist

Practical, copy-pasteable steps to ship **Distavo** (SwiftUI menu-bar app, `LSUIElement`,
macOS 14 deployment target) in its three editions:

| Edition         | Bundle ID                     | Signing                     | Sandbox | Channel                  |
| --------------- | ----------------------------- | --------------------------- | ------- | ------------------------ |
| **Direct**      | `uk.co.riera.distavo`         | Developer ID + notarized    | No      | GitHub / DMG             |
| **Setapp**      | `uk.co.riera.distavo-setapp`  | Developer ID + notarized    | No      | Setapp Marketplace       |
| **Mac App Store** | `uk.co.riera.distavo`       | Apple Distribution + profile | **Yes** | App Store                |

Project is generated by **XcodeGen** from `apple/project.yml`; per-edition build settings live in
`apple/configs/{Direct,Setapp,AppStore}.xcconfig`. Regenerate the Xcode project after any spec
change:

```bash
cd apple && xcodegen generate
```

> Convention used below: `<TEAM_ID>` = your 10-char Apple Developer Team ID, `<APPLE_ID>` =
> your Apple Account email, `<APP_PWD>` = an app-specific password. Replace placeholders.
> All `xcodebuild`/`codesign`/`xcrun` commands assume you run them from `/Users/marc/Distavo/apple`.

---

## 🚩 What's blocked on the developer (do these tomorrow, in order)

Everything below this banner needs **your** Apple / Setapp credentials. A bot cannot do it for you.

1. **[Apple ID + Team]** Confirm Team ID and that your Apple Developer Program membership is active
   (`https://developer.apple.com/account` → Membership). Plug Team ID into `DEVELOPMENT_TEAM` in
   `apple/project.yml` (currently empty) and regenerate.
2. **[Apple ID + Certificates]** Create / confirm two signing identities in your account:
   - **Developer ID Application** (for Direct + Setapp).
   - **Apple Distribution** + a **Mac App Store** provisioning profile for `uk.co.riera.distavo`
     (for the App Store edition).
3. **[Apple ID + App-specific password]** Generate an app-specific password
   (`https://account.apple.com` → Sign-In & Security → App-Specific Passwords) and run
   `xcrun notarytool store-credentials` once to create the **`distavo-notary`** keychain profile
   (see §1.3). This unblocks all notarization.
4. **[App Store Connect]** Create the app record for `uk.co.riera.distavo` (Direct/Setapp do **not**
   need this). Needed before any App Store upload (§4).
5. **[Setapp vendor account]** Email `developers@setapp.com` / sign up at
   `https://setapp.com/developers` to get a vendor account; first version must be uploaded via the
   Setapp **Web UI** (§3). Needs the Setapp Framework + public key integrated first.

Once 1–3 are done you can fully ship the **Direct** edition. 4 unblocks App Store; 5 unblocks Setapp.

## 1. One-time setup (credentials)

### 1.1 Certificates 🔑 **[Apple ID required]**

Easiest path is Xcode → **Settings → Accounts → Manage Certificates → +**:

- **Developer ID Application** — signs the Direct and Setapp builds for distribution outside the
  App Store. (There is also a *Developer ID Installer* cert; you only need it for `.pkg`/`.app`
  installers distributed outside the store — not required for a DMG/zip.)
- **Apple Distribution** — signs the Mac App Store `.app`.
- **3rd Party Mac Developer Installer** — signs the Mac App Store `.pkg` installer. A Mac App
  Store submission is a *signed installer package*, so this second identity is required in
  addition to Apple Distribution. Confirm both are present:
  `security find-identity -v -p codesigning`.

Then in your account portal create the **Mac App Store provisioning profile** for App ID
`uk.co.riera.distavo`, tied to the Apple Distribution cert. Download it (Xcode "automatic signing"
can also manage it, but this checklist uses **manual** signing to match `CODE_SIGN_STYLE: Manual`
in `project.yml`). The profile must be embedded in `Distavo.app`; otherwise App Store Connect can
accept delivery while marking the build ineligible for TestFlight with ITMS-90889.

Verify the identities are in your keychain:

```bash
security find-identity -v -p codesigning
# Expect to see:  "Developer ID Application: Joan Marc Riera Duocastella (<TEAM_ID>)"
#                 "Apple Distribution: Joan Marc Riera Duocastella (<TEAM_ID>)"
```

### 1.2 Hardened Runtime & entitlements (already wired)

All three xcconfigs set `ENABLE_HARDENED_RUNTIME = YES` (mandatory for notarization). The Direct
and Setapp editions use `apple/Distavo.entitlements` (effectively empty — **not** sandboxed). The
App Store edition uses `Distavo-AppStore.entitlements` (sandbox — see §4.2).

### 1.3 notarytool keychain profile 🔑 **[Apple ID + app-specific password required]**

Store credentials **once** so later commands don't need secrets inline. This writes to the login
keychain; the profile name `distavo-notary` is what every `notarytool` call references afterwards.

```bash
xcrun notarytool store-credentials "distavo-notary" \
  --apple-id "<APPLE_ID>" \
  --team-id "<TEAM_ID>" \
  --password "<APP_PWD>"          # the app-specific password, NOT your Apple account password
```

Sanity-check the stored profile:

```bash
xcrun notarytool history --keychain-profile "distavo-notary"
```

> Tip: an app-specific password is the simplest credential. If you'd rather use an App Store
> Connect API key, `store-credentials` also accepts `--key`, `--key-id`, and `--issuer`.

## 2. Direct edition (GitHub / DMG)

Developer ID signed + notarized, **not** sandboxed. This is the edition you can fully ship once
§1 is done.

### 2.1 Archive

```bash
xcodebuild -project Distavo.xcodeproj \
  -scheme Distavo \
  -configuration Release \
  -archivePath build/Distavo-Direct.xcarchive \
  archive \
  DEVELOPMENT_TEAM=<TEAM_ID> \
  CODE_SIGN_IDENTITY="Developer ID Application"
```

> The default scheme builds with the `Direct.xcconfig` (bundle `uk.co.riera.distavo`, hardened
> runtime on). If you maintain separate schemes per edition, pass the matching `-scheme`.

### 2.2 Export with the Developer ID method

Create `apple/ExportOptions-Direct.plist`:

```xml
<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE plist PUBLIC "-//Apple//DTD PLIST 1.0//EN" "http://www.apple.com/DTDs/PropertyList-1.0.dtd">
<plist version="1.0">
<dict>
  <key>method</key>            <string>developer-id</string>
  <key>teamID</key>            <string>TEAM_ID_HERE</string>
  <key>signingStyle</key>      <string>manual</string>
  <key>signingCertificate</key><string>Developer ID Application</string>
</dict>
</plist>
```

```bash
xcodebuild -exportArchive \
  -archivePath build/Distavo-Direct.xcarchive \
  -exportPath build/export-direct \
  -exportOptionsPlist ExportOptions-Direct.plist
# → build/export-direct/Distavo.app
```

### 2.3 Notarize 🔑 **[uses the keychain profile from §1.3]**

Notary service wants a zip/dmg/pkg, not a bare `.app`. Zip the app for submission (use `ditto` to
preserve symlinks/extended attributes):

```bash
ditto -c -k --keepParent build/export-direct/Distavo.app build/Distavo-Direct.zip

xcrun notarytool submit build/Distavo-Direct.zip \
  --keychain-profile "distavo-notary" \
  --wait
# --wait blocks until Apple returns Accepted/Invalid (usually a few minutes).
```

If status is `Invalid`, pull the log (replace `<id>` with the submission UUID it printed):

```bash
xcrun notarytool log <id> --keychain-profile "distavo-notary" developer_log.json
```

### 2.4 Staple

Staple the ticket to the `.app` so Gatekeeper passes even offline:

```bash
xcrun stapler staple build/export-direct/Distavo.app
xcrun stapler validate build/export-direct/Distavo.app   # → "The validate action worked!"
```

### 2.5 Wrap for distribution (DMG or zip)

**Zip** (simplest):

```bash
ditto -c -k --keepParent build/export-direct/Distavo.app build/Distavo.zip
```

**DMG** (nicer, draggable installer — recommended for GitHub Releases):

```bash
hdiutil create -volname "Distavo" \
  -srcfolder build/export-direct/Distavo.app \
  -ov -format UDZO build/Distavo.dmg
```

> Best practice: notarize and staple **the DMG too**, so the downloaded disk image itself is
> trusted. Re-run §2.3/§2.4 against `build/Distavo.dmg`:
>
> ```bash
> xcrun notarytool submit build/Distavo.dmg --keychain-profile "distavo-notary" --wait
> xcrun stapler staple build/Distavo.dmg
> ```

### 2.6 Verify (do this before you publish)

```bash
# Signature & hardened runtime are valid, recursively:
codesign --deep --verify --strict --verbose=2 build/export-direct/Distavo.app

# Show signing authority / that hardened runtime (flags=...runtime) is set:
codesign --display --verbose=4 build/export-direct/Distavo.app

# Gatekeeper assessment — the real "will it open on a clean Mac?" test:
spctl -a -vvv --type exec build/export-direct/Distavo.app
# Expect: "...accepted   source=Notarized Developer ID"
```

To simulate a fresh download (adds the quarantine flag) before testing on another Mac:

```bash
xattr -w com.apple.quarantine "0081;00000000;Safari;" build/export-direct/Distavo.app
```

### 2.7 Auto-update (optional) — Sparkle

For GitHub/DMG distribution, [Sparkle 2](https://sparkle-project.org/) is the standard auto-update
framework. Add it via SPM (`https://github.com/sparkle-project/Sparkle`), host an `appcast.xml`
(e.g. on GitHub Pages / a Release asset), and sign each update with your Sparkle EdDSA key
(`generate_keys` / `sign_update`). Sparkle is **only** for the Direct edition — Setapp ships its own
updater and the App Store updates through the store. Not required to launch.

## 3. Setapp edition

Setapp distributes Developer-ID-signed, **notarized**, **non-sandboxed** macOS apps through the
Setapp app. The build is essentially the Direct build with the `-setapp` bundle ID and the Setapp
Framework linked in. Confirmed against Setapp docs (2026); see Sources.

### 3.1 Vendor onboarding 🔑 **[Setapp vendor account required]**

- Sign up / request access at `https://setapp.com/developers` (or email `developers@setapp.com`).
- Setapp runs two models — know which you're in:
  - **Setapp Membership** (the bundle subscription): revenue is a **usage pool** — Setapp shares
    ~**70%** of subscription revenue across developers based on which apps each member *used* during
    the billing period, weighted by an app price-tier multiplier (1–100). You are paid for usage,
    not per-download. (+20% partner fee on users you referred.)
  - **Single-App Distribution** (newer, EEA + US only): sell Distavo as a one-time purchase or
    subscription via the Setapp Marketplace and keep **85%** of the app price.
- **The very first app version must be uploaded through the Setapp Web UI.** Later versions can be
  automated (REST API, the MacPaw shell-script template, or the Fastlane plug-in).

### 3.2 Technical requirements (build side)

1. **Bundle ID** must end in `-setapp`: `uk.co.riera.distavo-setapp`
   (already set in `apple/configs/Setapp.xcconfig`). Any helper executables become
   `uk.co.riera.distavo-setapp.<HelperName>`. **This ID is permanent — it can't be changed once set.**
2. **Notarized** (Apple notarization is required by Setapp).
3. **Setapp Framework** linked into the app. Add `libSetapp.a` and set in Build Settings /
   the xcconfig:

   ```
   OTHER_LDFLAGS = $(inherited) -force_load "$(BUILT_PRODUCTS_DIR)/libSetapp.a"
   ```

   Generate a **public key** in your Setapp vendor dashboard and add it to the app bundle, then
   activate Setapp at launch (gate behind the `EDITION_SETAPP` compilation condition that's already
   defined in `Setapp.xcconfig`, so the Direct/App Store builds never link Setapp). Follow the exact
   current API in `https://docs.setapp.com/docs/install-and-set-up-framework`.
4. **Info.plist** must include `CFBundleIdentifier`, `CFBundleName`, `CFBundleIconFile`,
   `CFBundleVersion`, `CFBundleShortVersionString`, and **`NSUpdateSecurityPolicy`** (required for
   macOS 13+). XcodeGen generates the Info.plist (`GENERATE_INFOPLIST_FILE: YES`), so add
   `NSUpdateSecurityPolicy` via an `INFOPLIST_KEY_…` setting or a custom plist.
5. **Bundle size** must not exceed **1 GB**.

### 3.3 Build, notarize, package

Archive + export + notarize + staple exactly as in §2.1–§2.4, but with the **Setapp** scheme/config
so the bundle ID is `uk.co.riera.distavo-setapp` and the Setapp Framework is linked.

Then package as the **zip** Setapp expects — a single root directory containing the `.app`, with
**no `__MACOSX`** metadata folder:

```bash
ditto -c -k --keepParent build/export-setapp/Distavo.app build/Distavo-Setapp.zip

# Setapp's own validation step — unpack and confirm one clean root dir, no __MACOSX:
ditto -x -k build/Distavo-Setapp.zip /tmp/distavo-check && ls -la /tmp/distavo-check
```

### 3.4 Submit 🔑 **[Setapp vendor account required]**

- **First version:** drag-and-drop `build/Distavo-Setapp.zip` into the **Build** area of the
  *Edit Version* page in the Setapp vendor dashboard, add release notes, set status to *review*.
- **Later versions:** automate via the Setapp REST API / MacPaw shell script / Fastlane plug-in.
- Setapp then reviews the build before it goes live.

Docs: `https://docs.setapp.com/` — start at *Single app distribution overview*,
*Set an app bundle ID*, *Install and set up Framework*, *Submitting apps for review*.

## 4. Mac App Store edition

Sandboxed, signed with **Apple Distribution** + a Mac App Store provisioning profile, uploaded to
App Store Connect, and put through App Review.

### 4.1 App Store Connect record 🔑 **[App Store Connect required]**

In `https://appstoreconnect.apple.com` → **Apps → +** create the app for bundle
`uk.co.riera.distavo` (register the App ID in the Developer portal first if it isn't there).
Fill in name, primary category, privacy policy URL, and the **App Privacy** questionnaire
(declare what Distavo collects — for a transcription/menu-bar app, likely "data not collected" or
microphone/audio usage; be accurate).

### 4.2 Sandbox entitlements + security-scoped bookmarks ⚠️

The App Store build **must** be sandboxed. `AppStore.xcconfig` already sets
`CODE_SIGN_ENTITLEMENTS = Distavo-AppStore.entitlements`. That file needs at minimum:

```xml
<key>com.apple.security.app-sandbox</key>                       <true/>
<key>com.apple.security.files.user-selected.read-write</key>    <true/>
<!-- add only what Distavo actually uses, e.g. -->
<key>com.apple.security.network.client</key>                    <true/>
<key>com.apple.security.device.audio-input</key>               <true/>   <!-- if it records audio -->
```

Because the sandbox forgets file access between launches, **any file/folder the user picks must be
persisted as a security-scoped bookmark** and re-resolved on next launch:

```swift
// Save (after the user picks a URL via NSOpenPanel):
let data = try url.bookmarkData(options: .withSecurityScope,
                                includingResourceValuesForKeys: nil, relativeTo: nil)
// Restore on next launch:
var stale = false
let url = try URL(resolvingBookmarkData: data, options: .withSecurityScope,
                  relativeTo: nil, bookmarkDataIsStale: &stale)
guard url.startAccessingSecurityScopedResource() else { /* handle */ }
defer { url.stopAccessingSecurityScopedResource() }
```

This is the single most common App Store regression vs. the Direct build — test it explicitly.

### 4.3 Screenshots & metadata 🔑 **[App Store Connect required]**

Provide macOS screenshots (Apple-accepted sizes, e.g. 2560×1600 / 2880×1800), description, keywords,
support URL, what's-new text, and a 1024×1024 app icon. A menu-bar app still needs representative
screenshots (show the menu/popover and any settings window).

### 4.4 Archive + upload

The `Distavo-AppStore` scheme (in `apple/project.yml`) pins `configs/AppStore.xcconfig` via its
`Release-AppStore` configuration, so the archive is the sandboxed edition. Use
**manual** signing and pin the Mac App Store provisioning profile. Apple can
accept a delivery that omits it, but reports ITMS-90889 and makes the build
ineligible for TestFlight when the main `.app` bundle lacks
`Contents/embedded.provisionprofile`.

Install the downloaded Mac App Store profile locally first:

```bash
PROFILE=~/Downloads/Distavo_AppStore.provisionprofile
PROFILE_PLIST=/tmp/distavo-appstore-profile.plist
security cms -D -i "$PROFILE" > "$PROFILE_PLIST"
PROFILE_UUID=$(/usr/libexec/PlistBuddy -c 'Print UUID' "$PROFILE_PLIST")
PROFILE_NAME=$(/usr/libexec/PlistBuddy -c 'Print Name' "$PROFILE_PLIST")
mkdir -p ~/Library/MobileDevice/Provisioning\ Profiles
cp "$PROFILE" ~/Library/MobileDevice/Provisioning\ Profiles/"$PROFILE_UUID".provisionprofile
```

```bash
xcodebuild -project Distavo.xcodeproj \
  -scheme Distavo-AppStore \
  -archivePath build/Distavo-AppStore.xcarchive \
  DEVELOPMENT_TEAM=<TEAM_ID> \
  CODE_SIGN_STYLE=Manual \
  CODE_SIGN_IDENTITY="Apple Distribution" \
  PROVISIONING_PROFILE_SPECIFIER="$PROFILE_NAME" \
  archive

test -f build/Distavo-AppStore.xcarchive/Products/Applications/Distavo.app/Contents/embedded.provisionprofile
```

Create `apple/ExportOptions-AppStore.plist` with the **`app-store-connect`** method and the two
signing identities pinned explicitly. (Pin them — a bare `signingStyle: automatic` export lets
xcodebuild's cert auto-selection grab the *installer* cert to sign the `.app` code, which fails with
"this identity cannot be used for signing code". The `.app` is signed by **Apple Distribution**, the
`.pkg` installer by **3rd Party Mac Developer Installer**.) Pin the provisioning
profile by UUID:

```xml
<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE plist PUBLIC "-//Apple//DTD PLIST 1.0//EN" "http://www.apple.com/DTDs/PropertyList-1.0.dtd">
<plist version="1.0">
<dict>
  <key>method</key>                     <string>app-store-connect</string>
  <key>teamID</key>                     <string>TEAM_ID_HERE</string>
  <key>signingStyle</key>               <string>manual</string>
  <key>signingCertificate</key>         <string>Apple Distribution</string>
  <key>installerSigningCertificate</key><string>3rd Party Mac Developer Installer</string>
  <key>provisioningProfiles</key>
  <dict>
    <key>uk.co.riera.distavo</key><string>PROFILE_UUID_HERE</string>
  </dict>
</dict>
</plist>
```

Export, then upload. Two equivalent options:

```bash
# Export the .pkg (needs the "3rd Party Mac Developer Installer" cert in your keychain):
xcodebuild -exportArchive \
  -archivePath build/Distavo-AppStore.xcarchive \
  -exportPath build/export-appstore \
  -exportOptionsPlist ExportOptions-AppStore.plist

# Verify the installer signature before uploading:
pkgutil --check-signature build/export-appstore/*.pkg

# Verify the exported payload still contains the provisioning profile:
pkgutil --payload-files build/export-appstore/Distavo.pkg | \
  grep -E '^(\./)?Distavo\.app/Contents/embedded\.provisionprofile$'

# Option A — upload from the command line (App Store Connect API key):
xcrun altool --upload-app -f build/export-appstore/Distavo.pkg \
  -t macos --apiKey <KEY_ID> --apiIssuer <ISSUER_ID>
# (xcrun notarytool is NOT used here — App Store builds are notarized by App Review.)

# Option B — drag the .pkg into Apple's Transporter.app (App Store Connect upload tool).
```

> App Store builds do **not** need the §2 `notarytool`/`stapler` steps — Apple notarizes during
> App Review. Sandbox + a correctly signed installer `.pkg` are what matter.

### 4.5 App Review 🔑 **[App Store Connect required]**

**Automated path (normal):** write `apple/metadata/whats-new/en-GB.txt`, push the `v*` tag, and
after the upload job succeeds press **Approve** on the `app-store-submission` environment in the
Actions run — `scripts/submit-appstore-review.py` then waits for build processing, sets export
compliance, attaches the build to the version (renaming the editable draft if needed), and submits
for review; the version releases automatically once Apple approves. Retry or stage-test any of
those steps with Actions → "Submit to App Review (manual)" (`dry_run` defaults on). Pricing /
availability / listing metadata still live in App Store Connect and only need setting once.

**Manual fallback:** in App Store Connect, attach the uploaded build to the version, set
pricing/availability, answer export-compliance, and **Submit for Review**.

Common rejections for a menu-bar app: missing/incorrect
sandbox entitlements, using private APIs, or a donate/payment link that bypasses Apple's IAP rules
(the `DONATE_ENABLED` flag is gated to the US storefront per `AppStore.xcconfig` — confirm that gating
holds at runtime before submitting).

**Built-in transcription engine + App Review:** downloading ML model weights at runtime is fine —
weights are data, not executable code (guideline 2.5.2 doesn't apply), Apple documents on-device
model download/compile for Core ML, and MacWhisper/Aiko ship exactly this pattern on the MAS.
Models land inside the sandbox container (`Application Support/Distavo/models`) via the existing
`network.client` entitlement — no new entitlements. In review notes, state that transcription is
fully on-device and the only network use is the user's own Ollama/WhisperX URLs plus the one-time
Hugging Face model download. Third-party licenses are in `NOTICES.md` (all MIT/Apache — no GPL).

## Quick verification cheat-sheet

```bash
codesign --deep --verify --strict --verbose=2 <App>     # signature OK, recursively
codesign --display --verbose=4 <App>                    # show authority + runtime flag
spctl -a -vvv --type exec <App>                         # Gatekeeper verdict
xcrun stapler validate <App|dmg>                        # ticket stapled
xcrun notarytool history --keychain-profile distavo-notary   # past submissions
```
